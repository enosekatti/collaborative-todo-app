## 1. Opening and Title (about 2–3 minutes)

"Good [morning/afternoon], everyone. Thank you for being here. Today I’m presenting the **Collaborative To-Do Application**, which I implemented in both **JavaScript** and **Java** as part of the Day 2 and Day 3 deliverables. The goal was to build the same application twice—with identical features and user experience—so we can compare how each language handles structure, concurrency, and persistence.

The full title of the project is **Collaborative To-Do Application: JavaScript and Java Implementations**. What we built is a **command-line to-do list** that supports **multiple users**. It isn’t just a single-user checklist; users can register, log in, and then create and manage their own tasks. Each task has a **title**, **description**, **category**, **status**, **priority**, and an optional **assignee**—so one person can create a task and assign it to someone else. The **categories** are fixed but extensible in code: Work, Personal, Shopping, Health, and Other. The **status** of a task goes through a simple lifecycle: Pending, In Progress, and Completed. Users can **filter** their view: they can list only their own tasks, only tasks in a given category, only tasks with a given status, or only tasks assigned to them. All of this is implemented in both the JavaScript version and the Java version, with the **exact same feature set** and the **same menu flow**, so that the comparison is fair and the focus is on how the two languages express the same design.

The source code lives in a **GitHub repository**; I’ll add the link on the title slide and in the written report. If anyone wants to run the application later, you can clone the repo and follow the README: the Node version runs with **npm start** from the project root, and the Java version runs from the **java** folder using the Maven Wrapper or Maven itself.

Next, I’ll walk you through the **architecture** that both implementations share, so you can see how we structured the application before we look at the actual code and demos."

## 2. Architecture — Detailed Walkthrough (about 4 minutes)

"The architecture is **intentionally the same** in both languages. We wanted to compare language features and idioms, not different designs. So in both the JavaScript and the Java codebase you’ll find the same layers and the same responsibilities.

At the **domain level** we have two main concepts: **User** and **Task**. The **User** module is responsible for user identity and profile. It defines what a user looks like—id, username, hashed password, display name, email, and creation timestamp—and it provides validation for username and password. For example, the username must be at least three characters and can only contain letters, numbers, underscore, and hyphen; the password must be at least four characters. The User module also exposes a **profile** view—that’s the same data but without the password hash, so we can safely show it in the CLI or in an API later. The **Task** module defines a task: id, title, description, category, status, priority, assignee, who created it, and created/updated timestamps. It provides validation for the title—required, max length 200—and for the whole task input, including that the category and status must be one of the allowed values and priority must be between zero and ten. So User and Task are the core data models and validation rules.

We also have **constant modules**: **Category** and **TaskStatus**. Category exports the five category strings—WORK, PERSONAL, SHOPPING, HEALTH, OTHER—and an immutable list of all of them, plus a function to check whether a string is a valid category. TaskStatus does the same for the three statuses: PENDING, IN_PROGRESS, COMPLETED. These constants are used everywhere we need to validate input or display options in the menu.

The **concurrency and persistence layer** is where we ensure that the application stays correct when multiple operations happen. We use a **Mutex**—a mutual exclusion lock. The rule is: any code that reads or writes the task or user data must do so **inside** the mutex. So only one such operation runs at a time. That prevents two operations from reading the same file, both modifying the data, and then both writing—which would cause lost updates or corruption. In JavaScript the mutex is **promise-based**: we don’t have real threads, but we do have async operations, so we built a lock that serializes those async operations by chaining Promises. In Java we use the standard **ReentrantLock** from java.util.concurrent.locks: the manager acquires the lock, does the work, and releases the lock in a finally block so it’s always released even if an exception occurs.

Under the mutex we have the **TaskRepository**. The repository is the **only** place that touches the file system. It knows where the data directory is, and it reads and writes **tasks.json** and **users.json**. In JavaScript the repository uses **fs.promises**—the promise-based file API—and we do atomic writes by writing to a temporary file and then renaming it to the real file, so a crash mid-write doesn’t leave a half-written file. In Java we use **NIO.2**—Files.readString and Files.writeString—and the same idea: write to a temp file, then move it over the target file. So the repository is responsible only for loading and saving; it doesn’t do validation or business logic.

On top of the repository we have **TaskManager** and **UserManager**. The **TaskManager** handles create, get by id, list with optional filters—category, status, assignee, or created-by—update, and delete. Every time it needs to read or write, it calls the mutex and inside the mutex it calls the repository. It also generates unique IDs for new tasks and validates the input using the Task module before creating or updating. The **UserManager** does the same for users: register, login, get by id, and update profile. Registration checks that the username isn’t already taken; login hashes the password and compares it to the stored hash. So the managers are the **orchestrators**: they enforce rules and call the repository under the lock.

The **user interface** is a **CLI**—a text menu. In JavaScript the entry point is **index.js**; in Java it’s **Main.java**. Both use a loop: show the menu, read the user’s choice, and dispatch to the right handler. When the user is **not** logged in, the menu shows: Register, Login, and Exit. When the user **is** logged in, it shows: List my tasks, List by category, List by status, List tasks assigned to me, Create task, Update task, Delete task, Logout, and Exit. So from the user’s perspective there is no difference between the two applications—same options, same prompts, same behavior. That’s the architecture. Now I’ll run the JavaScript version so you can see it in action."

## 3. Demo — JavaScript (about 4–5 minutes)

"Now I’ll run the **JavaScript** version. I’m in the project root directory—the folder that contains **package.json** and the **src** folder. I run **npm start**. That executes **node src/index.js**.

The application starts. The first thing we see is the line **Collaborative To-Do — CLI** and then the menu header **--- Collaborative To-Do ---**. Below that, because we’re not logged in, we see the hint **Log in to create and manage tasks**, then **1. Register**, **2. Login**, **0. Exit**, and the prompt **Choice:**. So the user types a number and presses Enter.

I’ll **register** first. I type **1** and Enter. The program then asks for **Username:**. I type **alice** and Enter. It asks for **Password:**. I type **test123**—the rule is at least four characters. Then it asks for **Display name (optional):** and **Email (optional):**. I’ll press Enter for both to skip them. The application validates the input, hashes the password, and writes the new user to **data/users.json**. Then it prints **Registered. You can now login.** and the menu appears again.

Now I **login**. I type **2** and Enter. It asks for **Username:** and **Password:**. I enter **alice** and **test123**. The application loads the users, finds the one with username alice, hashes the password I typed, and compares it to the stored hash. They match, so it prints **Logged in as alice**. The menu **changes**: we now see **Logged in as: alice** at the top, and the options are 1 through 8 for tasks and 0 to exit. So the CLI state has switched to the logged-in view.

I’ll **create a task**. I type **5** and Enter. It asks for **Title:**. I type **Finish report** and Enter. **Description (optional):** I press Enter to leave it blank. Then it prints the list of categories and asks **Category:**. I type **WORK** and Enter—it has to be uppercase to match our constants. **Priority (0-10, default 0):** I type **3** and Enter. **Assignee username (optional):** I press Enter to leave it blank. The TaskManager validates the input, generates a unique task ID—something like task_ and a timestamp and random suffix—creates the task object, and saves it to **data/tasks.json**. The program prints **Task created:** and the new task ID.

Now I’ll **list tasks** in different ways. Option **1** is **List my tasks**—tasks where I am the creator. I type 1 and Enter. It prints **My tasks (created by me):** and below that we see one line: the task ID in brackets, the title **Finish report**, the category **WORK**, the status **PENDING**, assignee **-** because there is none, and priority **3**. So that’s the task I just created. I go back to the menu. Option **2** is **List tasks by category**. I choose 2, and it shows the categories and asks **Category:**. I type **WORK** and Enter. We get the same task listed, because it’s the only WORK task. Option **4** is **List tasks assigned to me**. I choose 4. It prints **Tasks assigned to me:** and then **no tasks**, because we didn’t assign this task to anyone. So the filters are working: we can filter by creator, by category, and by assignee.

For **update** and **delete**: Option **6** is Update task. You’d enter the task ID—we could copy it from the list—then the program would show the current title, status, and category and ask for new values; you can press Enter to keep the current value. Option **7** is Delete task; you enter the task ID and it’s removed from the file. I won’t do those now to save time, but they work the same way in both apps. I’ll **logout** by typing **8** and Enter. It prints **Logged out.** and the menu returns to Register and Login. Then I type **0** to **Exit**. It prints **Bye.** and the process exits.

So that’s the full flow in JavaScript: register, login, create a task, list with different filters, and logout. Next I’ll do the same flow in Java so you can see that the behavior is identical."

## 5. Language-Specific Features and Code Comparison (about 4–5 minutes)

"Now I’ll describe how the two languages implement the same design differently—concurrency, data structures, asynchrony, and JSON handling.

**Concurrency.** In JavaScript the runtime is **single-threaded**. There are no real threads; we have an event loop. So when we do file I/O we use **async** functions so we don’t block the loop. But we still need to make sure that two async operations don’t read and write the file at the same time—because they could interleave and corrupt the data. So we built a **promise-based mutex**. The idea is: we keep a chain of Promises. When a new operation comes in, we create a new Promise and store its resolve function. The new operation first **awaits** the previous Promise in the chain, so it only runs after the previous one has finished. Then it runs its own async work—load tasks, maybe add one, save—and when it’s done it calls the resolve function so the next waiter can proceed. So we get **serialized** access to the critical section without blocking the event loop. In Java we have real threads, so we use the standard **ReentrantLock**. The TaskManager or UserManager calls **lock.lock()**, then in a try block it runs the critical section—load from repository, modify the list, save—and in a **finally** block it calls **lock.unlock()**. So only one thread can be inside that block at a time. The effect is the same: one operation at a time on the data; the mechanism is different—Promises in JavaScript, an explicit lock in Java.

**Data and types.** In JavaScript we use **plain objects** and **arrays**. There are no classes in our domain modules; we have factory functions like createUser and createTask that return objects with the right properties. For constants we use **Object.freeze** so the array of categories or statuses can’t be modified. When we **update** a task or user we don’t mutate the existing object; we return a **new** object using the **spread operator**—so we copy all the old fields and override only the ones that changed. That keeps the data flow predictable. In Java we use **final** classes and **final** fields. The Task and User classes have private final fields and getters; there are no setters. To “update” we have methods like **withUpdates** or **withProfile** that return a **new** instance with the changed fields. So both languages favor immutability; in Java the type system enforces it. For collections we use **List.of** in Java—that gives an unmodifiable list—for the list of categories and statuses. For operations that might not find something—like get task by ID—Java uses **Optional**. So getById returns **Optional of Task**; you call **isPresent** or **orElse** or **ifPresent** to handle the case when the task isn’t there. For operations that can fail with a message—like create or login—we use a small **Result** type: it has a boolean success, a value if successful, and an error string if not. So in JavaScript we return objects like **{ success: true, task }** or **{ success: false, error: '...' }**; in Java we return **Result.ok(task)** or **Result.fail("...")**. So JavaScript stays flexible with ad-hoc shapes; Java makes the success/failure and optional value explicit in the type system.

**Asynchrony and I/O.** In JavaScript, file operations are **asynchronous**. The TaskRepository has **async** functions. For example **loadTasks** is async; inside we **await** ensureDataDir and **await** readJson. So while we’re waiting for the disk, the event loop can do other work. That’s the standard Node style for I/O. In Java, I/O is **synchronous**. **loadTasks** is a normal method that **throws IOException**. We use **Files.readString** and **Files.writeString** from the NIO.2 API—they block until the read or write is done. So in JavaScript we get non-blocking I/O by default with async/await; in Java we have blocking I/O, and the mutex still protects the data if we later introduce multiple threads.

**JSON.** In JavaScript, **JSON is built in**. We use **JSON.parse** to read a string into an object and **JSON.stringify** to write an object to a string. No extra library. In Java there’s no JSON in the standard library, so we use **Gson**. We have small **entity** classes—TaskEntity and UserEntity—with a no-argument constructor and public fields so Gson can instantiate them and fill the fields from the JSON. When we load, we deserialize into a list of entities and then convert each entity to our domain Task or User. When we save, we convert each Task or User to an entity and then serialize the list. So the **data on disk** looks the same in both apps; the **code** that reads and writes it is different—native in JavaScript, Gson and entity classes in Java.

That’s the main comparison: same design, different language features and libraries."

## 6. Testing (about 2–3 minutes)

"We added **automated tests** in both languages to verify that the domain logic and validation work correctly.

In **JavaScript** we use Node’s **built-in test runner**. You run **npm test**, which runs **node --test** with our test files. The tests live under the **tests** folder. We have **category.test.js**: we check that ALL_CATEGORIES includes WORK, PERSONAL, OTHER and has length five, that isValidCategory returns true for valid categories like WORK and false for empty string, invalid string, or null. **TaskStatus** tests do the same for the three statuses. **User** tests: we test createUser—that it sets id, username, and createdAt; getProfile—that it returns an object without the passwordHash; validateUsername—that it rejects short or invalid usernames and accepts valid ones; validatePassword—that it requires at least four characters; and updateProfile—that it returns a new user with the updated display name or email. **Task** tests: we test createTask—defaults for status and priority; updateTask—that it returns a new task with the updated fields; validateTitle—rejects empty; validateTaskInput—rejects invalid category and accepts valid input. We don’t touch the file system in these tests; we’re only testing the pure domain and validation. When we run **npm test** we see all suites and tests pass—around seventeen tests in total.

In **Java** we use **JUnit 5**. We run **./mvnw test** or **mvn test**. Maven runs the Surefire plugin, which discovers test classes in **src/test/java**. We have **CategoryTest**, **TaskStatusTest**, **UserTest**, and **TaskTest**. They mirror the JavaScript tests: we assert that the constant lists contain the expected values and have the right size, that the validation methods return true or false as expected, that create and withUpdates or withProfile produce the right results, and that hashPassword is consistent for the same input. We don’t test the repository or the managers with real files in this suite; we rely on **manual testing** of the CLI for the full flow. So both codebases have a solid set of **unit tests** for the core domain and validation logic."

## 7. Challenges and Takeaways (about 2–3 minutes)

"I’ll briefly mention **challenges** we ran into and then **takeaways**.

In **JavaScript**, without static types we had to be disciplined. We used **JSDoc** to document the shape of objects and the return types of functions, so that we didn’t pass the wrong properties or forget to handle an error. The promise-based mutex was relatively straightforward to implement and fit naturally with async/await—we didn’t need any external library. In **Java**, we had more **boilerplate**: every domain class has multiple fields, a constructor, and getters; we have separate TaskEntity and UserEntity classes for Gson; and we have a small Result class. But we gained **compile-time checks**: if we pass the wrong type or forget to handle an Optional, the compiler tells us. **Streams** and **List.of** made the filtering and constant lists readable—we could express “filter tasks by category” as a stream pipeline instead of manual loops. So the trade-off is: JavaScript is faster to write and very flexible; Java is more verbose but catches more errors at compile time and makes the API contracts explicit.

**Takeaways.** Both implementations deliver the **same application**: user registration and login, full task CRUD, categories and statuses, filtered listing, and mutex-protected file storage so that concurrent access doesn’t corrupt the data. **JavaScript** is well-suited to quick iteration and to I/O-bound applications where async/await and the event loop are a natural fit; the downside is less static safety. **Java** gives strong typing, clear concurrency primitives like ReentrantLock, and good tooling for larger codebases; the downside is more boilerplate. The choice between them depends on your team’s experience, your deployment environment, and whether you prioritize rapid prototyping or long-term maintainability and type safety.
